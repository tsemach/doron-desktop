# Tag System Overhaul — Implementation Summary

## Goal

Replace the old plain-string tag arrays on cases/documents with a generic tag system that:
1. Distinguishes **system tags** (auto-generated by the app) from **user tags** (added manually).
2. Supports an **optional value** per tag (generalizing the old hardcoded `followup` + `followup_date` special case).
3. Is flexible enough to support future automated matching (e.g. an email-to-case classifier looking up a case by its `useremail` tag).

## Data model

New `tags` table (`apps/desktop/src-tauri/src/store/mod.rs`), replacing the `tags`/`followup_date` columns that used to live directly on `case_annotations`/`document_annotations`:

```sql
CREATE TABLE tags (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    scope_type  TEXT NOT NULL CHECK (scope_type IN ('case','document','app')),
    scope_value TEXT,              -- case id (as text) / document file_path / 'app' sentinel
    name        TEXT NOT NULL,
    value       TEXT,              -- optional
    type        TEXT NOT NULL CHECK (type IN ('user','system')),
    created_at  TEXT NOT NULL,
    updated_at  TEXT NOT NULL,
    UNIQUE(scope_type, scope_value, name)
);
```

- `scope_type='app'` is a singleton scope (fixed sentinel `scope_value`) for identity-level tags — `username`, `useremail` — that aren't tied to one case.
- A DB trigger (`tags_document_cascade`) deletes a document's tags when its `documents` row is hard-deleted. **Cases are only ever soft-deleted** (`delete_case` sets `deleted=1`, never a real `DELETE`), so no case-side cascade trigger exists — matches the existing `case_annotations` behavior, which has the same gap today.
- No data migration was needed/performed — the app isn't in production yet, so the legacy `tags`/`followup_date` columns were dropped outright rather than migrated.

## Backend API (`apps/desktop/src-tauri/src/tags/mod.rs`)

One generic layer instead of bespoke code per tag type:

- Internal (no IPC, used by other Rust modules): `upsert_tag_internal`, `remove_tag_internal`, `list_tags_internal`, `list_all_tags_for_scope_type`, `list_tags_for_document_fuzzy` (tolerant path matching for `list_case_files`), `find_by_name_value`.
- Tauri commands (frontend-facing, `type`/`scope_value` passed explicitly by the caller — not hardcoded, so the same commands cover both user and system tags): `add_tag`, `update_tag`, `remove_tag`, `list_tags`, `list_all_tag_names`.

### System tags wired up

- **`case_id`** — written when a case is created (`add_case`, `create_new_case` in `case/mod.rs`).
- **`username`** — written on `save_user_settings` (new `user_settings.rs` module + `user_settings` table, since the display name previously only lived in frontend `localStorage` with no backend source of truth).
- **`useremail`** — written on `save_email_settings` (`email/emails_settings.rs`), sourced from the IMAP config's `username` field (the configured mailbox address).

### Not built (intentionally out of scope)

`find_by_name_value` exists as the lookup hook for future automated matching (e.g. "find the case whose `useremail` tag equals this incoming email's sender") but nothing currently calls it — the email classifier itself wasn't part of this change, just the generic infrastructure it would need.

## Frontend

- `Tag` type + `TagChip` component (`components/ui/TagChip.tsx`) — system tags render muted/read-only with a lock icon; user tags are removable.
- All three tag-editing surfaces (`OpenCasesCaseAnnotationsModal`, `OpenCasesDocumentAnnotationsModal`, `OpenCasesUpdateDocumentModal`) and the read-only displays (`OpenCasesListItem`, `OpenCasesDocumentsPanelTopMenu`, `OpenCasesDocumentContent`) now use the generic tag commands instead of the old batched `tags: string[]` param on `set_case_annotations`/`set_document_annotations`.
- The `followup` tag is fully folded into the generic mechanism: typing "followup" as a tag name swaps in a date-picker for its value, and the previously-duplicated `getFollowupStatus` logic (present in two files) is now one shared `lib/followupStatus.ts`.
- `Settings.tsx` now persists the display name via `get_user_settings`/`save_user_settings` instead of `localStorage`.

## Verification

- `cargo check` and `npx tsc --noEmit` both pass clean.
- Drove the actual UI with a Playwright smoke test (mocked IPC layer) confirming: the `case_id` system tag renders correctly (muted, lock icon, non-removable), a new user tag can be added/removed, and typing "followup" correctly swaps in the date input.
